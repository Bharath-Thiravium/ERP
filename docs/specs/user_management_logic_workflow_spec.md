# Athens User Management Logic & Workflow Specification

## OVERRIDES (LATEST REQUIREMENT — MUST FOLLOW)
- Inside a project there is only ONE admin type: Project Admin (category can be Client/EPC/Contractor).
- Project Admin can create ONLY "Users" (employees) within their own scope (same project + same company).
- Project Admin must NOT create:
  - other Project Admins
  - any tenant-level admins / superadmins
  - any "Admin Users" (remove/ignore this concept if present in older code/spec sections)

Wherever this document mentions "Admin User", treat it as "User (Employee)" and enforce that Project Admin cannot create any admin roles.

---

## Overview
This document captures the business logic, user workflows, and screen content for Athens user management system. This specification is framework-agnostic and focuses purely on functional requirements.

---

## 1. USER HIERARCHY & TYPES

### Purpose
Multi-level user management system supporting project-based isolation with hierarchical user creation and management across different company types (Client, EPC, Contractor).

### User Types & Hierarchy

**Master Admin (Superuser)**
- `user_type` = 'master' OR `admin_type` = 'master'
- Top-level system administrator
- Can create projects and project admins
- Manages company details and logos
- Has access to all system functions

**Project Admin**
- `user_type` = 'projectadmin'
- `admin_type` = 'client', 'epc', or 'contractor'
- Created by Master Admin
- Manages users within their project/company scope
- Manages their own company details

**User (Employee)**
- Standard end user who uses the system features
- Must complete user profile details and get approval before full access
- (Replaces older "Admin User" concept)

---

## 2. ROLES & PERMISSIONS

### Master Admin Permissions
- Create projects and assign project admins
- View all projects across the system
- Reset passwords for any project admin
- Approve admin details for project admins
- Access consolidated dashboard data
- Manage menu configurations
- Delete projects (with dependency checks)

### Project Admin Permissions
- Create Users (employees) within their project AND company only
- View and manage users they created / within their scope
- Edit/delete users they created (and within scope)
- Reset passwords for users in their scope
- Approve user details for users in their scope
- View project-specific data only
- Update their own admin details

### User (Employee) Permissions
- Submit and update their own user details
- View their own profile information
- Cannot create or manage other users
- Cannot access user management screens

### Permission Rules
**Create User:** Project Admin can create Users (employees) in same project + same company  
**Edit User:** Creator OR Project Admin can edit when user in same project + company  
**Delete User:** Creator OR Project Admin can delete when user in same project + company  
**View Users:** Project Admin can view all users in their project + company  
**Approve Details:** Project Admin can approve user details in their scope  
**Password Reset:** Project Admin can reset passwords for users in their scope  

---

## 3. PROJECT ISOLATION

### Isolation Rules
- All users belong to exactly one project
- Users can only see/manage data within their project
- Master Admin can see all projects
- Project Admins see only their assigned project
- Users see only their project's data

### Company Inheritance
- Users inherit company information from their creator/scope
- Enforce: Project Admin creates users only within their own company

## 4. USER CREATION WORKFLOW

### A) Master Admin Creates Project Admin

**Required Fields:**
- `username` - Unique username
- `user_type` = 'projectadmin'
- `admin_type` - 'client', 'epc', or 'contractor'
- `project` - Project assignment
- `company_name` - Company name
- `registered_address` - Company address

**Optional Fields:**
- `email` - Email address
- `name` - First name
- `surname` - Last name

**Default Values:**
- `is_autogenerated_password` = true
- `is_password_reset_required` = true
- `is_active` = true
- Auto-generated 16-character password

**Process:**
1. Master Admin provides project and admin details
2. System generates secure password
3. Project Admin account created
4. Credentials provided to Master Admin
5. Project Admin must reset password on first login

### B) Project Admin Creates User (Employee)

**Required Fields:**
- `username` - Unique username (auto-filled from email)
- `email` - Email address
- `name` - First name
- `surname` - Last name
- `department` - Quality, Safety, Inventory, Project/Execution
- `designation` - Role designation
- `grade` - A, B, or C (auto-suggested based on designation)
- `phone_number` - 10-digit phone number
- `user_type` = 'user' (or equivalent employee type)

**Optional Fields:**
- Custom designation (can add new ones)

**Default Values:**
- `company_name` = creator's company_name
- `project` = creator's project
- `is_autogenerated_password` = true
- `is_password_reset_required` = true
- Auto-generated 16-character password

**Grade Assignment Rules:**
- Grade A: SiteIncharge
- Grade B: TeamLeader, Manager
- Grade C: All others

**Process:**
1. Project Admin fills user creation form
2. System auto-fills username from email
3. System suggests grade based on designation
4. System generates secure password
5. User account created with inherited project/company (must match creator scope)
6. Credentials downloaded as text file
7. New user must complete profile details

## 5. USER PROFILE DETAILS WORKFLOW

### Purpose
Users must complete detailed profile information for approval before full system access.

### A) User Detail Submission

**Personal Details (Required):**
- `name` - First name
- `surname` - Last name
- `gender` - Male, Female, Other
- `father_or_spouse_name` - Parent/spouse name
- `date_of_birth` - Must be 18+ years old
- `nationality` - Country of citizenship

**Official Details (Required):**
- `employee_id` - Unique employee identifier
- `education_level` - Educational qualification
- `date_of_joining` - Employment start date
- `mobile` - 10-digit phone number
- `mark_of_identification` - Physical identification mark

**Identity & Financial (Required):**
- `uan` - 12-digit UAN number
- `pan` - PAN format: ABCDE1234F (5 letters + 4 digits + 1 letter)
- `aadhaar` - 12-digit Aadhaar number

**Document Attachments (Required):**
- `photo` - Profile photo (upload or camera capture)
- `pan_attachment` - PAN card image/PDF
- `aadhaar_attachment` - Aadhaar card image/PDF

**Read-Only Fields:**
- `department` - From user account
- `designation` - From user account
- `email` - From user account

**Validations:**
- Name/Surname: Letters and spaces only
- Phone/UAN/Aadhaar: Digits only, exact length
- PAN: Specific format validation
- Date of Birth: Must be 18+ years
- File uploads: Images or PDF, size limits

### B) Profile Photo Capture
**Methods:**
- File upload (drag/drop or browse)
- Camera capture using webcam
- Photo preview and replacement

**Requirements:**
- Required field for all users
- Image formats supported
- File size validation
- Camera permission handling

### C) Digital Signature Template
**Purpose:** Auto-generated signature template for document signing
**Process:**
- Template created automatically when profile submitted
- Uses user's name, surname, designation
- Stored for reuse across documents

## 6. APPROVAL WORKFLOW

### A) User Detail Approval Process

**Approval Authority:**
- Project Admin approves User (employee) details
- Master Admin approves Project Admin details

**Approval States:**
- `is_approved` = false (default, pending approval)
- `is_approved` = true (approved)

**Process:**
1. User submits complete profile details
2. System sends notification to approver
3. Approver reviews submitted details
4. Approver can approve or request changes
5. Upon approval, user gets access (with induction gating)
6. System sends approval notification to user
