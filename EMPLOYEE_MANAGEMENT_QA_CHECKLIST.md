# Employee Management Workflow - Manual QA Checklist

## Pre-requisites Setup
- [ ] Master Admin account exists and can login
- [ ] At least one Company is created and approved
- [ ] At least one Athens Project is created
- [ ] Project Admin account is created and linked to project/company

## Test Scenario 1: Project Admin Creates Employee

### 1.1 Login as Project Admin
- [ ] Login with Project Admin credentials
- [ ] Verify dashboard loads correctly
- [ ] Verify user type is identified as 'project_admin'

### 1.2 Create Employee
- [ ] Navigate to Employee Management
- [ ] Click "Create Employee" button
- [ ] Fill required fields:
  - [ ] Email: `testemployee@company.com`
  - [ ] First Name: `Test`
  - [ ] Last Name: `Employee`
  - [ ] Department: `Quality`
  - [ ] Designation: `Engineer`
  - [ ] Grade: `C`
  - [ ] Phone: `1234567890`
- [ ] Click "Create Employee"
- [ ] Verify success message appears
- [ ] Verify credentials are returned in response
- [ ] Verify credentials file downloads automatically
- [ ] Verify employee appears in employee list with status "not_started"

### 1.3 Verify Scope Enforcement
- [ ] Verify created employee has same project as Project Admin
- [ ] Verify created employee has same company as Project Admin
- [ ] Verify created_by field is set to Project Admin user
- [ ] Verify employee has `is_password_reset_required=true`
- [ ] Verify employee has `is_autogenerated_password=true`

## Test Scenario 2: Employee First Login & Password Reset

### 2.1 Employee First Login Attempt
- [ ] Logout from Project Admin account
- [ ] Try to login with employee email and temporary password
- [ ] Verify login succeeds but redirects to password reset page
- [ ] Verify access to other pages is blocked

### 2.2 Password Reset Process
- [ ] On password reset page, enter:
  - [ ] Email: `testemployee@company.com`
  - [ ] Old Password: `[temporary password from credentials]`
  - [ ] New Password: `NewSecurePass123!`
- [ ] Click "Reset Password"
- [ ] Verify success message
- [ ] Verify `is_password_reset_required` is set to `false`
- [ ] Verify automatic redirect to profile completion page

### 2.3 Verify Access Stage Progression
- [ ] Call `/me/access-state/` endpoint
- [ ] Verify stage is now `must_complete_profile`
- [ ] Verify allowed_menu_keys contains `['profile', 'password_reset']`
- [ ] Try to access other pages - verify blocked with redirect

### 3.1 Profile Completion Form
- [ ] Verify profile completion form loads

- [ ] Fill Step 1 - Personal Details:

  - [ ] Nationality: `Indian`

  - [ ] Education Level: `Bachelor's Degree`

### 3.3 Identity & Financial Details
- [ ] Fill Step 3:
  - [ ] UAN: `123456789012`
  - [ ] PAN: `ABCDE1234F`
  - [ ] Aadhaar: `123456789012`
- [ ] Click "Next" - verify Step 4 loads

### 3.4 Document Uploads
- [ ] Upload Step 4 documents:
  - [ ] Profile Photo: Upload valid image file
  - [ ] PAN Attachment: Upload valid image/PDF
  - [ ] Aadhaar Attachment: Upload valid image/PDF
- [ ] Verify "Submit for Approval" button is enabled
- [ ] Click "Submit for Approval"
- [ ] Verify success message
- [ ] Verify redirect to pending approval page

### 3.5 Verify Profile Submission
- [ ] Verify employee `profile_status` is `submitted`
- [ ] Verify `profile_submitted_at` timestamp is set
- [ ] Call `/me/access-state/` - verify stage is `pending_approval`
- [ ] Verify access to other pages is blocked

## Test Scenario 4: Project Admin Approval Process

### 4.1 Login as Project Admin
- [ ] Logout from employee account
- [ ] Login as Project Admin
- [ ] Navigate to Employee Management
- [ ] Verify "Pending Approvals" section shows 1 employee
- [ ] Verify employee appears with status "submitted"

### 4.2 Review and Approve
- [ ] Click "Review" button for pending employee
- [ ] Verify approval modal opens with employee details
- [ ] Click "Approve"
- [ ] Verify success message
- [ ] Verify employee disappears from pending approvals
- [ ] Verify employee status changes to "approved" in main list

### 4.3 Verify Approval Effects
- [ ] Verify employee `profile_status` is `approved`
- [ ] Verify `profile_approved_at` timestamp is set
- [ ] Verify `profile_approved_by` is set to Project Admin

## Test Scenario 5: Induction Training Access

### 5.1 Employee Login After Approval
- [ ] Logout from Project Admin
- [ ] Login as employee with new password
- [ ] Call `/me/access-state/` - verify stage is `approved_but_induction_pending`
- [ ] Verify allowed_menu_keys contains `['profile', 'training', 'induction']`
- [ ] Try to access other modules - verify blocked

### 5.2 Access Induction Training
- [ ] Navigate to induction/training page
- [ ] Verify induction modules are displayed
- [ ] Verify only training-related pages are accessible
- [ ] Complete at least one mandatory induction module
- [ ] Verify module completion is recorded

### 5.3 Complete All Induction
- [ ] Complete all mandatory induction modules
- [ ] Verify `has_completed_induction` is set to `true`
- [ ] Verify `induction_completed_at` timestamp is set
- [ ] Call `/me/access-state/` - verify stage is `full_access`
- [ ] Verify allowed_menu_keys contains `['all']`

## Test Scenario 6: Full Access Verification

### 6.1 Full Portal Access
- [ ] Verify employee can now access all allowed modules
- [ ] Verify no more redirects or access blocks
- [ ] Verify employee workflow is complete

## Test Scenario 7: Scope Enforcement Tests

### 7.1 Project Admin Scope Limits
- [ ] Login as Project Admin
- [ ] Verify employee list shows only employees from same project/company
- [ ] Verify cannot see employees from other projects/companies
- [ ] Verify pending approvals only shows employees in scope

### 7.2 Cross-Company Access Test
- [ ] Create second company and project (as Master Admin)
- [ ] Create second Project Admin for different company
- [ ] Create employee under second Project Admin
- [ ] Login as first Project Admin
- [ ] Verify cannot see employees from second company
- [ ] Verify cannot approve employees from second company

## Test Scenario 8: Rejection Workflow

### 8.1 Create and Reject Employee Profile
- [ ] Create new employee (as Project Admin)
- [ ] Complete employee profile submission process
- [ ] As Project Admin, reject the profile with reason
- [ ] Verify employee status changes to "changes_requested"
- [ ] Verify rejection reason is stored

### 8.2 Employee Resubmission
- [ ] Login as rejected employee
- [ ] Verify redirected back to profile completion
- [ ] Verify can edit and resubmit profile
- [ ] Verify resubmission changes status back to "submitted"

## Test Scenario 9: API Security Tests

### 9.1 Unauthorized Access Tests
- [ ] Try to access employee management APIs without authentication
- [ ] Verify 401 Unauthorized responses
- [ ] Try to access with wrong user type
- [ ] Verify 403 Forbidden responses

### 9.2 Scope Violation Tests
- [ ] Try to create employee in different project (modify payload)
- [ ] Verify request is rejected
- [ ] Try to approve employee from different company
- [ ] Verify request is rejected

- [ ] Verify timestamps are accurate
- [ ] Verify status transitions are logged

### 10.2 Migration Test
- [ ] Run database migrations on clean database
- [ ] Verify all tables are created correctly
- [ ] Test backfill migration with existing data
- [ ] Verify existing users are properly migrated
